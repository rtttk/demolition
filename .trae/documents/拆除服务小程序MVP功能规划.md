# 拆除服务小程序 MVP 功能规划

## 一、项目背景与任务理解

### 1.1 任务重述
- **目标**：为上海本地一家80人规模的拆除公司设计一款微信小程序MVP，聚焦"小微拆除"场景
- **产品形态**：**一个小程序 + 一个Web后台**。小程序内通过角色区分（需求方/服务方）展示不同功能视图，平台运营使用独立的Web后台管理系统
- **组织结构**：**公司为主体，团队为细分**。资质认证和合规信息挂在**公司**维度（法律责任主体），**团队**是公司内部的施工班组，团队维度负责接单和施工执行。MVP阶段一个需求由一个团队承接，不做多团队协作
- **核心价值**：建立安全、绿色、合规的拆除全生命周期解决方案，解决零散客户发布需求难、寻找合规拆除团队难的痛点
- **商业模式**：信息整合 + 资源匹配的双边撮合平台（需求方 ↔ 拆除服务方）
- **输出**：MVP功能规划文档，明确最小可行产品应包含的功能模块

### 1.2 市场现状分析
| 维度 | 现状 | 机会 |
|------|------|------|
| **需求侧** | 零散客户（业主、小商户、物业）拆除需求碎片化，信息不对称严重，难以找到合规团队 | 缺乏垂直领域的专业拆除信息平台 |
| **供给侧** | 拆除公司获客依赖传统渠道（人脉、58同城），信任成本高，资质展示不充分 | 合规资质难以在通用平台上有效传达 |
| **政策侧** | 2025年7月起《上海市既有建筑物、构筑物拆除工程管理规定》实施，要求开工备案、资质审核、文明施工等 | 合规门槛提高，非正规队伍被淘汰，合规团队价值凸显 |
| **竞品侧** | 58同城/赶集网等通用平台信息杂乱、无专业审核；万师傅/鲁班到家聚焦家居安装维修，未覆盖拆除领域 | 拆除垂直领域存在平台空白 |

### 1.3 核心痛点梳理
- **客户侧**：① 不知道去哪发布拆除需求 ② 无法判断拆除团队是否合规/有资质 ③ 报价不透明 ④ 施工过程缺乏安全保障 ⑤ 建筑垃圾清运无保障
- **拆除团队侧**：① 获客渠道单一、成本高 ② 小微项目利润薄但流程复杂 ③ 资质和能力难以展示 ④ 客户信任度低

## 二、MVP 功能规划

### 2.1 MVP 核心原则
> **"最小功能集验证最大假设"** — MVP应验证的核心假设是：零散客户是否愿意通过小程序发布拆除需求，拆除团队是否愿意通过平台抢单接单，以及施工过程透明化是否能有效提升客户信任和验收体验。

- 聚焦上海本地，不做跨区域
- 先做信息撮合，不做深度交易闭环（支付、合同在线签署等延后）
- **公司为主体，团队为细分**：资质认证和合规信息在公司维度，团队是公司下属施工班组，负责接单和施工
- 合规审核前置，建立平台信任壁垒
- **抢单制 + 平台报价审核**：需求公开发布，团队主动抢单报价，平台审核后展示给客户
- **施工过程透明化**：每日汇报日志，让客户实时了解施工进度，方便验收
- **单团队承接**：MVP阶段一个需求由一个团队承接，不做多团队协作（数据模型预留扩展空间）
- 以"小微拆除"场景切入（室内拆除、店面拆除、围墙拆除、小型构筑物拆除等）

```
┌──────────────────── 微信小程序 ────────────────────────┐
│  用户登录后根据角色（需求方/服务方）展示不同功能视图       │
├──────────────┬────────────────────────────────────────┤
│  需求方视图    │  服务方（公司→团队）视图                  │
├──────────────┼────────────────────────────────────────┤
│ · 需求发布    │ · 公司信息管理（资质认证/合规信息）         │

│ · 公司/团队浏览│ · 抢单大厅（浏览/筛选需求）                │
│ · 案例浏览    │ · 在线报价（以团队身份）                   │
│ · 施工日志查看 │ · 每日施工日志上传（文字/图片/视频）         │
│ · 留言板沟通   │ · 留言板沟通                              │
│ · 施工验收    │ · 接单/施工状态跟踪                        │
│ · 评价        │ · 查看评价                                │
└──────────────┴────────────────────────────────────────┘

┌──────────────────── Web后台管理系统 ────────────────────┐
│  平台运营人员使用                                        │
├────────────────────────────────────────────────────────┤
│ · 用户管理（需求方/服务方）                               │
│ · 公司资质审核（营业执照/拆除资质/安全生产许可证）          │
│ · 团队管理（查看公司下属团队）                             │
│ · 需求审核                                              │

│ · 合规模板文档维护                                       │
│ · 施工日志监控                                          │
│ · 数据采集与分析                                        │
│ · 投诉/纠纷处理                                         │
└────────────────────────────────────────────────────────┘
```

### 2.3 功能详细说明

#### 模块一：需求发布（需求方视图）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **快速需求发布** | 引导式表单，3步完成发布：①选择拆除类型 → ②填写基本信息（地址、面积、描述）→ ③上传现场照片 | P0 |
| **拆除类型分类** | 室内拆除/店面拆除/围墙拆除/小型构筑物拆除/管道拆除/其他，每种类型提供专属描述模板 | P0 |
| **智能估价参考** | 基于拆除类型+面积+楼层等维度，给出价格区间参考（非精确报价），降低客户心理门槛 | P1 |
| **图片/视频上传** | 支持上传现场照片（最多9张），帮助拆除团队初步判断 | P0 |
| **地址定位** | 调用微信地图API自动定位，支持手动修正，精确到门牌号 | P0 |
| **需求状态跟踪** | 客户可查看需求状态：待抢单→报价审核中→已推荐报价→已选择团队→施工中→待验收→已完成 | P0 |

#### 模块二：公司与团队管理（服务方视图）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **公司资质认证** | 公司维度上传营业执照、拆除资质证书、安全生产许可证等，平台审核后展示公司认证标识 | P0 |
| **公司信息展示** | 展示公司信息：公司名称、成立时间、资质等级、合规状态、累计完成项目数 | P0 |
| **团队管理** | 公司下可创建多个施工班组（团队），每个团队有独立信息：团队名称、负责人、人员规模、擅长类型、服务区域 | P0 |
| **团队主页** | 展示团队信息：所属公司、团队规模、擅长类型、累计完成项目数、平均评分 | P0 |
| **案例展示墙** | 以团队维度上传过往拆除案例（施工前后对比图），增强信任感，客户可浏览所有团队案例 | P0 |
| **评价体系** | 客户完工后对团队评分+文字评价，展示在团队主页，作为客户选团队的核心参考依据 | P0 |
| **接单大厅** | 展示平台上的需求列表，团队可浏览、筛选、抢单报价 | P0 |
| **在线报价** | 团队对感兴趣的需求提交报价方案（含价格、工期、施工方案简述） | P0 |

#### 模块三：抢单报价机制（服务方视图 + Web后台）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **抢单大厅** | 所有合规团队均可看到平台上的新需求，团队主动浏览、筛选、抢单报价 | P0 |
| **在线报价** | 团队对感兴趣的需求提交报价方案（含价格、工期、施工方案简述） | P0 |
| **平台报价审核** | 平台运营人员对团队提交的报价进行审核，确保报价合理性和合规性，审核通过后展示给客户 | P0 |
| **多报价对比展示** | 客户可同时看到多个通过审核的报价方案，对比价格、工期、团队资质等信息后选择 | P0 |
| **地域/类型/资质筛选** | 团队端可按区域、拆除类型筛选需求列表，平台端按资质匹配需求与团队 | P0 |
| **人工辅助干预** | 平台运营人员可手动调整匹配结果、催促报价、标记优质团队，积累数据优化算法 | P1 |

#### 模块四：订单管理（需求方 + 服务方共用）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **需求-报价跟踪** | 客户查看收到的报价列表，团队查看已报价需求的反馈 | P0 |
| **留言板式沟通** | 报价审核通过后，客户和团队可在订单详情页互相发送留言消息（文字+图片），非实时类似留言板，沟通记录保留在平台便于追溯 | P0 |
| **联系方式展示** | 报价审核通过后展示双方联系电话，方便紧急情况直接电话沟通 | P0 |
| **状态流转** | 需求状态自动流转：发布→抢单→报价审核→报价展示→选择团队→施工→待验收→完工 | P0 |
| **完工确认** | 客户确认完工，触发评价流程 | P0 |

#### 模块五：施工过程跟踪（需求方 + 服务方共用）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **每日汇报日志** | 施工团队每日上传施工日志，记录当天工作内容、进度、人员安排等 | P0 |
| **多媒体记录** | 支持文字描述 + 图片上传（施工前后对比）+ 视频上传，完整记录施工过程 | P0 |
| **施工进度透明化** | 客户可实时查看每日施工日志，了解当天做了什么、拆除了哪些部分 | P0 |
| **验收成果记录** | 施工日志形成完整的施工档案，客户可基于日志逐项验收，方便确认成果 | P0 |
| **日志时间线** | 按日期展示施工日志时间线，直观呈现整个施工过程的全貌 | P0 |
| **异常情况记录** | 支持记录施工中遇到的异常情况（如发现隐蔽管线、结构问题等），及时通知客户 | P1 |

#### 模块六：基础支撑模块（小程序 + Web后台）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **用户体系** | 微信授权登录，区分"需求方"和"服务方"两种角色，同一用户可切换角色，不同角色展示不同功能视图 | P0 |
| **消息通知** | 微信模板消息/订阅消息推送：新需求通知、新报价通知、状态变更通知、施工日志更新通知 | P0 |
| **合规模板库** | 内置上海拆除工程合规知识库：备案流程指引、所需材料清单、政策法规速查；支持后台手工维护文档内容 | P0 |
| **数据采集** | 采集核心业务数据：用户行为、需求发布、报价响应、施工日志、评价等，为后续分析看板提供数据基础 | P0 |
| **后台管理** | Web端管理后台：用户管理、需求审核、团队资质审核、报价审核、合规模板文档维护 | P0 |
| **数据分析看板** | 核心指标可视化：发布需求数、报价响应率、成交转化率、平均工期、完工率、客户满意度等 | P1 |

## 三、MVP 版本功能优先级矩阵

### P0 — 必须有（MVP核心）
1. 微信授权登录 + 角色区分
2. 需求发布（引导式表单 + 图片上传 + 地址定位）
3. 公司资质认证 + 公司信息展示 + 团队管理（公司→团队两级结构）
4. **案例展示墙**（施工前后对比图，增强信任）
5. **用户评价体系**（评分+文字评价，客户选团队核心参考）
6. 抢单大厅 + 在线报价 + 平台报价审核
7. 多报价对比展示
8. 留言板式沟通 + 联系方式展示
9. 订单状态流转 + 完工确认
10. **施工过程跟踪**（每日汇报日志 + 文字/图片/视频记录 + 进度透明化 + 验收成果记录）
11. 消息通知（微信订阅消息，含施工日志更新通知）
12. 合规模板库（含后台文档手工维护功能）
13. 数据采集（核心业务数据埋点）
14. Web后台管理（用户/公司资质/团队/需求/报价审核 + 合规模板文档维护）

### P1 — 应该有（增强体验）
1. 智能估价参考
2. 人工辅助匹配/干预
3. 异常情况记录
4. 数据分析看板（核心指标可视化）

### P2 — 暂不做（后续迭代）
1. 在线支付/担保交易
2. 电子合同签署
3. 施工过程直播/远程监控
4. 建筑垃圾清运对接
5. 保险购买对接
6. 拆除物资回收交易
7. 拼团/集采模式
8. **合规模板库 — 资讯信息模块**（行业资讯、政策解读等资讯类内容）

## 四、关键页面流程

### 4.1 客户端核心流程
```
首页 → 点击"发布需求" → 引导式表单填写(3步)
→ 提交成功 → 等待匹配（需求进入抢单大厅，所有合规团队可见）
→ 平台审核报价 → 展示通过审核的报价列表给客户
→ 收到报价 → 查看报价详情（对比价格/工期/团队资质/评价）→ 选择团队
→ 在线沟通（留言板+电话） → 确认合作 → 状态跟踪
→ 施工过程跟踪（查看每日施工日志、图片/视频记录）
→ 施工验收 → 用户评价
```

### 4.2 拆除团队端核心流程
```
首页(接单大厅) → 浏览需求列表 → 筛选/搜索
→ 查看需求详情 → 提交报价
→ 与客户沟通（留言板+电话） → 等待确认
→ 确认接单 → 施工执行 → 每日上传施工日志（文字/图片/视频）
→ 完工 → 获得评价
```

### 4.3 平台运营流程
```
后台登录 → 审核新团队资质 → 审核需求信息
→ 监控抢单报价情况 → 审核团队报价（合理性与合规性）
→ 监控施工日志上传情况 → 处理投诉/纠纷
→ 维护合规模板文档 → 数据采集与分析
```

## 五、技术方案建议

### 5.1 技术选型
| 层面 | 推荐方案 | 理由 |
|------|---------|------|
| **小程序端** | uni-app | 需求方和服务方共用一个小程序，通过角色区分不同视图；跨平台能力，后续可扩展H5/App |
| **后台端** | Vue3 + Element Plus | 平台运营人员使用的Web后台管理系统 |
| **后端** | Node.js (Nest.js) | 高效开发，适合I/O密集型业务，与前端同语言降低协作成本 |
| **数据库** | MySQL + Redis | 结构化数据存储 + 缓存 |
| **地图** | 腾讯地图微信小程序SDK | 与微信生态无缝集成 |
| **对象存储** | 阿里云OSS | 存储图片/视频/文档，成本可控，CDN加速 |

### 5.2 合规要点（基于上海2025年新规）
根据《上海市既有建筑物、构筑物拆除工程管理规定》（2025年7月1日实施）：
- 平台应在需求发布环节提示客户：拆除工程需办理开工备案
- 团队入驻时必须先注册公司，公司维度验证：营业执照、资质等级证明、安全生产许可证
- 平台应提供合规模板：施工组织方案模板、备案所需材料清单
- 信息展示：明确区分"需备案项目"和"小微免备案项目"的界限

## 六、冷启动策略建议

### 6.1 供给侧先行
- **自有团队入驻**：80人公司作为种子服务方，确保初期有足够的承接能力
- **同行邀请**：邀请上海本地3-5家合规拆除公司入驻，丰富供给侧
- **资质背书**：突出平台"合规认证"标签，区别于58同城的野蛮生长

### 6.2 需求侧获取
- **线下场景**：物业公司合作（老小区改造、商铺装修拆除）、装修公司异业合作
- **线上推广**：微信朋友圈广告（上海地域定向）、小红书"拆除避坑"内容营销
- **口碑裂变**：完工客户分享评价到朋友圈，获取新客户

### 6.3 信任建立
- 每个入驻团队必须通过资质审核，展示认证标识
- 平台承诺：所有团队均持有效资质，如发现违规全额退款
- 施工前后对比图案例展示，增强客户信心

## 七、风险与应对

| 风险 | 应对策略 |
|------|---------|
| **双边冷启动** | 供给侧先行，自有团队保障初期服务能力 |
| **非标服务定价难** | MVP阶段提供价格区间参考，不承诺精确报价 |
| **合规风险** | 平台仅做信息撮合，明确免责条款；资质审核前置 |
| **低频需求** | 拓展拆除上下游（如垃圾清运、回收），提高用户粘性 |
| **线下履约质量** | 评价体系 + 平台抽查 + 退出机制 |

## 十、总结

本MVP方案围绕**"合规拆除信息撮合平台"**定位，以上海本地小微拆除场景为切入点，通过以下核心功能验证市场假设：

1. **需求发布** — 降低客户发布拆除需求的门槛
2. **公司资质认证** — 公司维度建立合规信任壁垒，区别于通用分类信息平台
3. **团队管理** — 公司下属施工班组独立管理，灵活承接不同项目
4. **抢单报价机制** — 团队主动抢单 + 平台审核报价，保障报价质量与合规性
4. **案例展示墙 + 评价体系** — 施工前后对比案例 + 真实用户评价，构建选队决策依据
5. **施工过程跟踪** — 每日汇报日志（文字/图片/视频），施工进度透明化，方便客户验收
6. **留言板式沟通 + 联系方式展示** — 轻量沟通方案，保留平台沟通记录，支持电话直联
7. **订单管理** — 完整的需求生命周期管理（发布→抢单→报价审核→施工→验收→评价）
8. **单团队承接** — MVP阶段一个需求由一个团队承接，数据模型预留多团队协作扩展空间

MVP阶段不做深度交易闭环，聚焦验证"需求是否存在"和"撮合是否有效"两个核心假设，同时通过施工过程跟踪功能建立服务透明化壁垒，为后续迭代奠定基础。
